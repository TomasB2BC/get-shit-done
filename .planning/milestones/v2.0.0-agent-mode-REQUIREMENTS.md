## v2.0.0 Requirements

Requirements for autonomous agent-mode milestone. Each maps to roadmap phases.

### Dispatcher & Orchestration (7/7)

- [x] **DISP-01**: /gsd:auto command invokes auto-dispatch workflow for autonomous pipeline
  - Outcome: auto.md command stub invokes auto-dispatch.md workflow
- [x] **DISP-02**: Auto-dispatch reads STATE.md and determines next action (plan/execute/verify/complete)
  - Outcome: Dispatcher reads current_phase + plan status from STATE.md to determine action
- [x] **DISP-03**: Auto-dispatch spawns fresh Task agent per phase (Chain-of-Agents pattern)
  - Outcome: Task tool invoked per action with full context prompt, preventing context degradation
- [x] **DISP-04**: Dispatcher stays thin (<10k tokens per cycle) -- never does heavy work itself
  - Outcome: Dispatcher only reads STATE.md and dispatches -- no plan/research file reading
- [x] **DISP-05**: Dispatcher updates STATE.md after each phase completes
  - Outcome: STATE.md phase counter and status updated after each Task completion
- [x] **DISP-06**: Dispatcher detects milestone completion and terminates cleanly
  - Outcome: All-phases-complete check triggers clean exit with summary
- [x] **DISP-07**: Opus 1M context: dispatcher can hold multi-phase context without /clear when feasible
  - Outcome: Model preference in Task spawning (Opus 1M preferred, Sonnet fallback)

  - Outcome: Rule engine with per-type defaults, overridable via config
- [x] **AUTO-03**: config.json agent_mode field toggles between classic and agent mode
  - Outcome: agent_mode boolean in config.json (defaults false)
- [x] **AUTO-04**: config.json agent_mode_settings section for autonomy level and rule overrides
  - Outcome: agent_mode_settings with autonomy_level, auto_scope, max_iterations
- [x] **AUTO-05**: Every auto-decision logged to AUTO-DISPATCH-LOG.md with decision + rationale
  - Outcome: cmdLogDecision writes timestamped entries with tiered verbosity
- [x] **AUTO-06**: Agent mode is opt-in (agent_mode defaults to false, classic behavior preserved)
  - Outcome: All agent-mode branches gated behind config check, classic paths unchanged

### Lead-Approval Integration (6/6)

- [x] **LEAD-01**: Decision classification taxonomy (architectural vs operational)
  - Outcome: Dispatcher-action-level classification (generate-context/re-plan/halt = architectural; others = operational)
- [x] **LEAD-02**: Architectural decisions routed to human via AskUserQuestion
  - Outcome: AskUserQuestion with classification rationale, context, and response options
- [x] **LEAD-03**: Lead can approve or reject with feedback; agent continues accordingly
  - Outcome: Approve continues, reject triggers 1 revision round, double-reject halts with HALT.md
- [x] **LEAD-04**: Session-death recovery via write-ahead PENDING_APPROVAL.md; no traditional timeout (AskUserQuestion blocks until response; halt on errors)
  - Outcome: PENDING_APPROVAL.md written before AskUserQuestion, checked on --resume
- [x] **LEAD-05**: Three autonomy levels configurable: auto-decide, lead-approval, full-auto
  - Outcome: autonomy_level in agent_mode_settings with /gsd:settings UI
- [x] **LEAD-06**: Lead-approval active only when agent_mode=true AND autonomy_level=lead-approval
  - Outcome: Compound condition checked in dispatcher classification logic

- **SAFE-04**: Fail-fast validation at phase boundaries
- **SAFE-05**: Stuck loop detection (no STATE.md change in N cycles = abort) -- implemented with guard
- **SAFE-06**: State validation assertions after each phase

### Recursive GSD

- **RGSD-02**: Autonomy depth tracking passed to recursive invocations
- **RGSD-03**: Lead-teammate coordination (assign phases, report progress, review verification)
- **RGSD-04**: Deadlock prevention (circular dependency detection, timeouts)

### Advanced

- **ADV-01**: Conservative auto-scoping defaults (table stakes only unless depth=comprehensive)

- **ADV-03**: Synthetic task testing before production deployment

## Out of Scope

| Feature | Reason |
|---------|--------|
| Replace executor with Agent Teams | Deterministic wave execution needs Task subagents |
| Nested teams | Agent Teams doesn't support teammates spawning sub-teams |
| Skip verification in agent mode | Defeats GSD core value (plan-execute-verify discipline) |
| "God Agent" single-context pipeline | Causes context degradation -- Chain-of-Agents solves this |
| Database or message queue | File-based coordination (STATE.md) is canonical |
| External dependencies | GSD stays markdown-native with zero npm dependencies |
