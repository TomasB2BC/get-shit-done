**Step 5: Pass 4 -- Boundaries**

```
>> Pass 4: Boundaries
>> "What does this govern? What doesn't it touch?"
```

Sub-steps:

1. Based on the decision understanding from Passes 1-3, generate a set of 5 edge-case questions. Each tests scope: "Does this also apply to X?" or "Does this override Y?" or "What happens when Z?"

Format (same pattern as Pass 3):
```
## Boundaries: Question Set {round}

**Q1: [Edge case question about scope]**
  a) [Pre-digested answer -- applies to this case]
  b) [Pre-digested answer -- does NOT apply to this case]
  c) [Pre-digested answer -- partially applies with conditions]
  d) Write your own

[Q2-Q5 same format]

---
>> Which questions would you like to answer? (e.g., "1a, 2b, 3c" or "all")
>> Type "clear" if the boundaries are already obvious.
```

2. Parse responses. Update scope understanding.

3. Loop: If human wants more edge cases, generate another round. Maximum 3 rounds for boundaries (tighter than Deep Dig's 5 rounds).

4. After sufficient rounds or "clear" signal, produce the scope statement:
```
>> Scope Statement:
   This decision governs:
   - [X]
   - [Y]
   - [Z]

   This decision does NOT govern:
   - [A]
   - [B]
   - [C]

>> Confirm scope or adjust?
```

5. Human confirms or adjusts. Store final scope statement for Pass 6 crystallization.

**Design notes:**
- Same question-set UX as Pass 3 (lettered options, select-which-to-answer)
- Focus is narrower: scope boundaries, not decision content
- Prevent over-application: the scope statement explicitly lists what is OUT
- Maximum 3 rounds (vs Deep Dig's 5) because boundaries are more constrained

**Step 6: Pass 5 -- Stress Test**

```
>> Pass 5: Stress Test
>> "The strongest case against this decision"
```

Sub-steps:

1. Based on everything gathered in Passes 1-4, construct the strongest counterargument. Display it clearly:
```
>> The best case against this decision:
   [2-3 paragraph counterargument -- genuine, not strawman]
```

2. Present 5 attack-angle questions with lettered options:
```
## Stress Test: Attack Angles

**Q1: Contradiction** -- Does this contradict any existing decisions?
  a) [Pre-digested: specific contradiction identified]
  b) [Pre-digested: no contradiction, compatible because...]
  c) [Pre-digested: partial tension but manageable because...]
  d) Write your own assessment

**Q2: Scalability** -- Will this hold up as the project grows?
  a) [Pre-digested: scales well because...]
  b) [Pre-digested: will need revision at [threshold]]
  c) [Pre-digested: does not scale, but acceptable for current scope]
  d) Write your own assessment

**Q3: Dependency Risk** -- What breaks if a dependency changes?
  a) [Pre-digested: low risk because...]
  b) [Pre-digested: specific dependency risk identified]
  c) [Pre-digested: mitigated by...]
  d) Write your own assessment

**Q4: Opportunity Cost** -- What are we giving up by choosing this?
  a) [Pre-digested: alternative we are foregoing]
  b) [Pre-digested: minimal opportunity cost because...]
  c) [Pre-digested: significant cost but justified because...]
  d) Write your own assessment

**Q5: Second-Order Effects** -- What downstream changes does this force?
  a) [Pre-digested: specific downstream effect]
  b) [Pre-digested: no significant downstream effects]
  c) [Pre-digested: downstream effects that need mitigation]
  d) Write your own assessment

---
>> Answer all 5 or select which matter most.
>> The decision either survives stronger or gets modified.
```

3. Parse responses. If any attack angle reveals a genuine weakness:
```
>> This challenge has merit. Options:
   a) Modify the decision to address it
   b) Accept the risk with documented rationale
   c) Revisit Pass 3 with this new constraint
```

4. After stress test complete:
```
>> Stress test complete. Decision [survived intact | was modified].
>> Proceeding to crystallization.
```

**IMPORTANT:** Always run the full stress test. No --trusted flag, no shortcuts, no skipping. Even well-debated decisions benefit from one adversarial pass.

**Step 7: Pass 6 -- Crystallization + Close**

```
>> Pass 6: Crystallization + Close
>> "Record, propagate, commit"
```

Sub-steps:

1. Present the final decision statement (1-3 sentences, crisp):
```
>> Final Decision Statement:
   [1-3 sentence crystallized decision]
```

2. Present the structured record:
```
## Decision Record: [Name]

**Statement:** [1-3 sentences -- what we decided]
**Rationale:** [What informed it -- research, discussion, evidence from Passes 1-5]
**Provenance:** Session [date], during [context -- phase discussion, research review, etc.]
**Scope:**
  - Governs: [from Pass 4 scope statement]
  - Does NOT govern: [from Pass 4 scope statement]
**Connections:** [Extends/modifies which existing decisions, if any]
**Date:** [YYYY-MM-DD]
```

3. Human confirms or adjusts. This is the final shape of the decision.

4. **Edit Target Discovery** -- Spawn 3-5 parallel Explore recon probes to discover which files need updates:

```
Task(
  prompt="You are an edit-target recon agent. Scan .planning/ files for sections that reference or are affected by this decision domain:

Decision: [decision statement]
Scope: [governs / does not govern]

Scan [specific subset of files] for:
- Sections that discuss the same domain
- Tables with related entries
- Claims that the decision supersedes or modifies

For each potential edit target, output:
TARGET: [file path]
SECTION: [heading or table name]
CURRENT: [what it currently says, brief]
PROPOSED: [what it should say after this decision]
CONFIDENCE: [HIGH | MEDIUM | LOW]

Write to: .scratch/elevate/pass6-targets-{N}.md",
  subagent_type="Explore",
  description="Edit target discovery: [file subset]"
)
```

Suggested probe distribution:
- Probe A: PROJECT.md (Key Decisions table, kitchen model if present)
- Probe B: ROADMAP.md + STATE.md (phase descriptions, accumulated context)
- Probe C: REQUIREMENTS.md + active PLAN.md files (requirements and plan assumptions)
- Probe D: MEMORY.md + any other .planning/ root files (if they exist)

5. Read all probe outputs. Deduplicate targets. Present with lettered options per target:
```
## Edit Targets

**Target 1: PROJECT.md -- Key Decisions table**
  Current: [no entry for this decision]
  Proposed: Add row: | [date] | [name] | [1-sentence summary] |

  a) Apply this edit
  b) Skip this target
  c) Adjust the edit (describe what to change)

**Target 2: STATE.md -- Accumulated Context**
  Current: [no reference to this decision]
  Proposed: Add to Decisions section: "[decision name] -- [brief summary]"

  a) Apply this edit
  b) Skip this target
  c) Adjust the edit

[Additional targets...]

---
>> Select for each target (e.g., "1a, 2a, 3b, 4c-[your adjustment]")
```

6. Execute all approved edits:
- For each "a" (apply): Read target file, apply the proposed edit using Edit tool. CRITICAL: Match the target file's existing style -- column widths, heading format, list format, voice, detail level. Read the surrounding content first to calibrate.
- For each "c" (adjust): Apply with the human's modification.
- Track all changed files.

7. Clean up `.scratch/elevate/pass6-targets-*.md`

8. Git commit with provenance:
```bash
git add [all changed files]
git commit -m "docs: elevate decision -- [decision name] (source: [session context])"
```

9. Display completion and rollback offer:
```
>> Decision crystallized and propagated.
   Changed {N} files: [list]
   Commit: [hash]

   To undo: git revert HEAD
```

**IMPORTANT design rules for Pass 6:**
- Edit targets are DISCOVERED by recon probes, not hardcoded
- Each edit is STYLE-MATCHED: AI reads existing rows/sections and follows same format
- MEMORY.md edits require explicit human approval (never auto-apply)
- Commit message MUST follow format: docs: elevate decision -- [name] (source: [context])
- Probe files go to .scratch/elevate/ and are cleaned up
- All options use letters (a, b, c, d), never numbers
